# Python コードレビュー・スキル

Python（このリポジトリは **3.12+ / uv workspace**）の差分に対する**言語固有**の観点。
一般観点・severity 語彙（`[must]`/`[imo]`/`[nits]`/`[ask]`/`[fyi]`）は [`code-review-general`](../code-review-general/SKILL.md)、
領域境界（`public.py`・`.importlinter`）は [`code-review-architecture`](../code-review-architecture/SKILL.md) を併せて使う。

> **設計根拠**: [ADR-0004 PRレビューをエージェント＋観点別スキルで運用する](../../../docs/adr/0004-pr-review-agent.md)

> severity の原則は general と同じ：**正確性・明示要件・セキュリティに効くものだけ `[must]`**。スタイルや好みは `[nits]`/`[imo]`。

---

## 観点カテゴリ（チェックリスト）

### 1. 型と契約
- [ ] 公開関数・メソッド・`public.py` の関数に**型ヒント**があるか（引数・戻り値）。`[must]`: 公開IFが型不明
- [ ] `Optional`/`None` の扱いが明示的か。`None` が返り得る関数を非 Optional として扱っていないか
- [ ] 可変デフォルト引数（`def f(x=[])` / `={}`）を使っていないか（`[must]`: 典型的バグ）
- [ ] `Any` の濫用や、実態と食い違う型注釈がないか

### 2. 例外・エラー処理
- [ ] `except Exception:` / 裸の `except:` で握りつぶしていないか。**根本原因に対処**しているか（`[must]`: 握り潰し）
- [ ] 例外を握って `None`/空を返して症状を隠していないか。失敗は呼び出し元に伝わるか
- [ ] 例外の種類が適切か（広すぎ/狭すぎ）。メッセージに原因が分かる情報があるか
- [ ] `raise ... from e` で原因を連鎖しているか（必要な箇所で）

### 3. 設計・可読性
- [ ] 関数・クラスが**単一責務**か。長すぎる関数を分割すべきでないか
- [ ] エンティティ/値オブジェクトに `@dataclass`（必要に応じ `frozen=True`）など適切な道具を使っているか
- [ ] 早期 return でネストを浅くできないか
- [ ] マジックナンバー/文字列を定数化すべきでないか（`[nits]`〜`[imo]`）

### 4. I/O・副作用の境界
- [ ] ビジネスロジック（domain）に DB/HTTP/ファイル/時刻/乱数などの副作用が混入していないか（アーキタイプA）。`[must]`: domain の純粋性違反
- [ ] 副作用は端（adapters）に寄っているか
- [ ] グローバル可変状態・モジュールレベルの副作用に依存していないか

### 5. リソース管理
- [ ] ファイル/ソケット/ロック等を `with`（コンテキストマネージャ）で確実に解放しているか（`[must]`: リーク）
- [ ] 例外時にもリソースが解放されるか

### 6. テスト（pytest）
- [ ] 変更に見合う pytest があるか。**コードが壊れたら失敗する**テストか（`[must]`: 常にパスする飾りテスト）
- [ ] アサーションが具体的で意味があるか。1テスト1観点に分かれているか
- [ ] 起こり得ないケースの過剰なテストを足していないか（過剰指摘の抑制と同じ精神）
- [ ] テスト名が「何を確認するか」を表しているか

### 7. 依存・標準ライブラリ
- [ ] 標準ライブラリで足りるものに外部依存を増やしていないか（`[imo]`〜`[ask]`）
- [ ] 依存を追加する場合、`pyproject.toml` の適切な場所（領域パッケージ / dev グループ）に入っているか
- [ ] uv workspace の作法（領域は `members`、領域間は `public.py`）に反していないか → architecture スキル

### 8. 命名・スタイル
- [ ] 変数/関数/モジュールは `snake_case`、クラスは `PascalCase`、定数は `UPPER_SNAKE` か
- [ ] 命名が説明的で、周辺コードと一貫しているか
- [ ] スタイル指摘は `[nits]`。機能変更と混ぜず分離を提案する

## severity の目安（Python 特有）

| `[must]` になりやすい | `[imo]`/`[nits]` に留める |
|----------------------|--------------------------|
| 可変デフォルト引数・例外の握り潰し・リソースリーク | 命名の好み・コメントの体裁 |
| domain への I/O 混入（純粋性違反） | マジックナンバーの定数化 |
| 常にパスするテスト・テスト欠如（要件にテストが必要な場合） | 内部関数の型ヒント追加提案 |
| `None` 取り違えによる実バグ | 早期 return への整形 |

## 確認コマンド（証拠主義）

レビュー対象ブランチで以下を確認できると指摘の確度が上がる。

```bash
uv run pytest           # テストが実際に通る/落ちることを確認
uv run lint-imports     # 領域境界（.importlinter）。境界違反は architecture スキル参照
```

> 「直った/動く」の主張ではなく、上記の**出力**で判断する（[`code-review-general`](../code-review-general/SKILL.md) の証拠主義）。
