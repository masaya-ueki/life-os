# スライド構成スキル

プレゼンテーションの**ストーリー設計**を担うスキル。「何を・どの順番で・なぜ伝えるか」を決め、章立て（chapters）と各スライドの役割に落とし込む。表現方法（図表の作り方）は [`slide-expression`](../slide-expression/SKILL.md) スキルが担当する。本スキルは **構成（順序と論理）** に集中する。

---

## 設計の大原則

1. **課題・目的(Why)を必ず最初に立てる**: 「何の課題を、なぜ解くのか」が不明なプレゼンは刺さらない。表紙の直後で課題と目的を提示する。
2. **1スライド1メッセージ**: 1枚のスライドで主張は1つ。`summary`（概要）がそのスライドの主張＝結論になる。`content`（内容）はその根拠・詳細。
3. **結論先行（トップダウン）**: 各章・各スライドは「結論 → 根拠」の順。聞き手は結論を先に知りたい。
4. **情報量の上限**: 1スライドの箇条書きは原則 3〜5 項目、1行は短く。多いなら分割する。
5. **流れに山場を作る**: 平坦に説明せず、「課題の深刻さ → 転換（解決策）」で緩急をつける（起承転結の「転」）。

---

## 1. 起承転結のプレゼン適用型

物語の型をプレゼンに適用する。技術・提案系プレゼンの王道。

| 段 | 役割 | プレゼンでの中身 | 対応する章 |
|----|------|-----------------|-----------|
| **起** | 導入・前提共有 | 背景・テーマの提示。「今こういう状況です」 | 背景 |
| **承** | 課題の深掘り | 何が問題か、放置するとどうなるか。痛みを具体化 | 課題/懸念 |
| **転** | 転換・解決策 | 「そこで、こうします」。提案・対策の核心（山場） | 目的・提案・対策 |
| **結** | 効果と結論 | 解決後の世界・効果・まとめ・次アクション | 効果・まとめ |

> ポイント: 「承（課題）」を十分に重くしてから「転（解決策）」に移ると、解決策の価値が際立つ。

---

## 2. 課題・目的(Why)を明確化する型

冒頭で「なぜこの話を聞くべきか」を作る代表的フレーム。テーマに応じて選ぶ。

### SCQA（おすすめ・課題提起に最適）
- **S**ituation（状況）: 共有できる前提。「Claude Code の利用が広がっている」
- **C**omplication（複雑化・問題）: 状況を揺るがす事実。「だが任意コード実行のリスクがある」
- **Q**uestion（問い）: 自然に湧く問い。「どう安全に使うか？」
- **A**nswer（答え）: 本プレゼンの主張。「権限設計・サンドボックス・レビューで対策できる」

### PREP（1スライド/結論先行の単位に最適）
Point（結論）→ Reason（理由）→ Example（具体例）→ Point（再結論）。各スライド内の論理構造としても使う。

### ゴールデンサークル（Why-How-What）
Why（なぜやるか・目的）→ How（どうやるか・方針）→ What（具体策）。**目的から語る**ことで共感を作る。提案・ビジョン系に強い。

### Problem-Solution / Before-After
課題（Problem / Before）と解決後（Solution / After）を対で見せる。効果が直感的に伝わる。→ 表現は `comparison`。

### ピラミッドストラクチャー（ミント）
頂点にメインメッセージ、その下に根拠を MECE に並べる。全体が「結論→根拠→詳細」の階層になる。アジェンダ全体の論理チェックに使う。

---

## 3. 定番アジェンダ構成（ビジネス/技術プレゼン）

迷ったらこの並び。各ブロックが1〜複数スライド（= chapters の各 chapter）に対応する。

```
1. 表紙        title        … タイトル・サブタイトル・日付/発表者
2. アジェンダ   bullet       … 本日の流れ（章の一覧）
3. 背景        bullet       … 起。前提・コンテキスト共有
4. 課題/懸念   structure    … 承。問題の構造化・深刻さ（comparison/structure）
5. 目的        emphasis     … 転の入口。何を達成するのか（Why の言語化）
6. 提案/対策   flow         … 転。解決策の核心（flow/comparison）
7. 詳細        comparison   … 各対策の具体・比較・根拠
8. 効果        emphasis     … 結。Before-After・数値・KPI
9. まとめ      bullet       … 要点の再掲（3点に圧縮）
10. Next Action bullet      … 次の一歩・依頼事項
```

> 短いプレゼンは 3/5 を統合、4 を1枚に圧縮するなど調整する。**表紙・課題・対策・まとめ**は省略しない4本柱。

## 4. 構成スキル → outline.yml へのマッピング

設計した構成を `domains/presentation/README.md` 定義の YAML スキーマに落とす。1 ブロック = 1 `chapter`、1 主張 = 1 `slide`。

```yaml
deck:
  title: "..."        # 表紙のメインメッセージ
  subtitle: "..."
  theme: default
chapters:
  - chapter: "背景"           # アジェンダのブロック名（起承転結の段でも可）
    slides:
      - title: "..."          # スライド見出し
        summary: "..."        # ★そのスライドの主張＝結論（1メッセージ）
        content: ["...", "..."]  # 根拠・詳細（3〜5項目）
        expression: bullet    # 表現（slide-expression の早見表で選ぶ）
        data: {}              # 表現固有データ（comparison/chart/flow/structure 用）
```

落とし込みの手順:
1. テーマから**メインメッセージ**（deck.title が表す主張）を1文で決める。
2. 起承転結 or 定番アジェンダで**章（chapters）**を並べる。各章が論理の1ステップ。
3. 各章を**1主張=1スライド**に割る。`summary` に結論、`content` に根拠を書く。
4. 各スライドの内容に最適な `expression` を [`slide-expression`](../slide-expression/SKILL.md) の早見表で選ぶ。
5. 全体を**ピラミッド**で検算: 各 `summary` を上から読んで筋が通るか確認する。

---

## チェックリスト（構成完成の判定）

- [ ] 表紙の直後で**課題・目的**が提示されている
- [ ] 各スライドが **1メッセージ**（summary が結論として独立して読める）
- [ ] 章が**起承転結 or 結論→根拠**の論理順に並んでいる
- [ ] 「転（解決策）」の前に「承（課題）」で十分に痛みを描いている
- [ ] **まとめ**で要点が3点程度に圧縮されている
- [ ] 各 `summary` を上から読むだけでストーリーが通る（ピラミッド検算）
