# 用途理解型の設問設計と誤答づくり

暗記でなく「**使いどころの理解**」を測る問題を作るための指針。

## 用途理解型とは

知識の再生（用語・数値の暗記）ではなく、**要件・目的からの選択・判断**を問う。良い問いは「状況（要件）」を示し、「それを満たすために何を使う／どうするのが適切か」を選ばせる。

### 設問の型（いずれかに寄せる）

| 型 | 骨子 | 例 |
|---|---|---|
| 要件充足型 | 「〜を満たすためには何が必要か／どれを使うべきか」 | 「取り込み済みデータを誤って削除した。**追加コストなく**復元するには何を使うべきか？」→ Time Travel |
| 目的達成型 | 「〜を実現する最も適切な機能／設定はどれか」 | 「本番テーブルの巨大コピーを**ストレージを二重消費せず**即時に用意したい。適切な方法は？」→ Zero-copy Clone |
| 理由選択型 | 「なぜ X が適切か／不適切か」 | 「点在した小さな更新を高頻度で流し込む。Snowpipe より Snowpipe Streaming が適する理由は？」 |
| 誤り診断型 | 「この状況で性能が出ない原因はどれか」 | 「クエリが遅く Query Profile で bytes spilled to remote storage が大きい。まず取るべき対処は？」 |

### アンチパターン（避ける）

- 用語の定義当て（「Time Travel とは何か」）。
- 数値・上限の丸暗記（「Standard の Time Travel 最大保持日数は？」のような単独暗記）。ただし**要件に絡めて**問うなら可（「90 日の保持が必要。必要なエディションは？」）。
- 選択肢が本文の言い換えで一意に決まる、常識で消せる。
- 「すべて正しい／どれも正しくない」を正解にする。

## 選択肢（distractor）の作り方

- 誤答は**もっともらしい**こと。よくある誤解・隣接概念・似た機能を当てる（例: Fail-safe と Time Travel、Snowpipe と Snowpipe Streaming、Materialized View と Result Cache）。
- 4件は**同じ粒度・同じカテゴリ**で揃える（機能名なら全部機能名）。
- 位置の偏りは出題側でシャッフルされるが、`is_correct` はデータ上の論理値。正解の位置に意味を持たせない。

### `ng_reason`（全選択肢必須）

- 誤答: 「なぜ不正解か」を一次情報ベースで簡潔に。単に「誤り」ではなく**どう誤っているか**（役割違い・条件不足・別機能）を書く。
- 正解肢: 「正解。〜」で始め、なぜ正しいかを一言添える（既存データの慣習）。
- 例（誤答）: 「Fail-safe は Snowflake 運用者による災害復旧用で、ユーザーが任意に復元する手段ではない。」

## `explanation`

- 正解の**核心**を1〜2文で。要件と機能の対応（「〜という要件には〜が適する」）を明示すると学習効果が高い。

## single / multiple の使い分け

- `single`: 最適解が1つに定まる問い（要件充足・目的達成の多く）。正解ちょうど1件。
- `multiple`: 「**当てはまるものをすべて選べ**」と明示し、複数の正しい選択肢がある問い。正解2件以上。採点は**完全一致**（部分点なし）なので、正誤の境界が明確なトピックに限る。
