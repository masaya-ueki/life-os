Claude Code とは

Claude Code はターミナルで動き、ファイル編集やコマンド実行までエージェントが代行する開発ツール。

  • 自然言語の指示でコード探索・編集・テスト・コミットを実行
  • ファイル読み書き・シェルコマンド・MCP 経由の外部ツール連携が可能
  • 強力ゆえに『何をどこまで実行させるか』の制御が前提になる
  • 利用が広がるほど、組織としての安全な運用設計が必要

懸念の全体像

リスクは『影響の大きさ × 起こりやすさ』で4象限に整理でき、まず右上から手当てする。

→ 影響の大きさ → 起こりやすさ
サプライチェーン・外部MCP汚染（要備え）
破壊的変更 / 任意コマンド実行（最優先）
軽微な情報露出（監視で対応）
操作ミス・誤差分（仕組みで防ぐ）

懸念の核心: 漏えいとインジェクション

破壊的操作に加え、シークレットの読み取りと悪意あるコンテンツによる指示の乗っ取りが二大経路。

信頼できない入力 → 被害
機密情報の漏えい
シークレット読取(.env/鍵)
ログ・履歴への露出
プロンプトインジェクション
悪意あるファイル/README
汚染された依存・外部MCP

本プレゼンの目的

便利さを殺さず、取り返しのつかない事故をゼロに

生産性を落とさずに『取り返しのつかない事故』をゼロに近づけることがゴール。

対策の全体方針

『権限で絞る → 環境で隔離する → 人がレビューする』の三層で多重防御する。

  1. 1

    権限で絞る

    許可制 / プランモード

  2. 2

    環境で隔離

    サンドボックス / コンテナ

  3. 3

    人がレビュー

    差分・コミット前確認

対策1: 権限モードを使い分ける

既定は許可制かプランモードにし、全自動の自動承認は限定された安全な環境だけに留める。

プランモード 許可制(既定推奨) 自動承認
実行前の確認 計画のみ提示 都度確認 なし
破壊的操作 実行しない 確認後のみ そのまま実行
適する場面 設計・調査 通常の開発 隔離環境限定

対策2: シークレット管理と隔離

シークレットは『読ませない・置かない・隔離する』を徹底し、流出経路を物理的に断つ。

  • 読ませない: .env / secrets を deny ルールと除外設定で対象外に
  • 置かない: 鍵はリポジトリ外、.gitignore でコミット防止
  • 隔離する: サンドボックス/コンテナで本番認証情報に触れさせない
  • 最小権限: トークンは用途を絞った短命なものを使う

対策3: レビューと信頼できないコンテンツの扱い

差分は必ず人がレビューし、信頼できない入力はデータとして扱い指示にしない。

  • 差分レビュー: 変更は人が確認してからコミット/プッシュ
  • コミット前確認: force push・本番反映は手動ゲートを置く
  • 信頼できないコンテンツ: 外部Issue/Web/MCP応答を『命令』として実行しない
  • 自動レビュー: 公式 security-guidance プラグイン等で変更を機械チェック（要確認: 提供状況）

万一に備える: インシデント対応フロー

防御をすり抜けても、検知から復旧・再発防止まで回せる運用を用意しておく。

  1. 1

    検知

    監査ログ・差分監視

  2. 2

    封じ込め

    権限剥奪・トークン失効

  3. 3

    復旧

    Git から安全状態へ復元

  4. 4

    再発防止

    ルール・手順を更新

対策導入の効果(Before / After)

三層防御で『取り返しのつかない事故』を仕組みで止め、安心して活用できる状態になる。

Before（無防備）

  • 全許可で破壊的操作が自動実行
  • シークレットが読み取り可能
  • 差分レビューは任意
  • 汚染入力を指示として実行
→

After（三層防御）

  • 許可制/プランモードで都度確認
  • シークレットは読ませない・隔離
  • コミット前レビューを必須化
  • 信頼できない入力はデータ扱い

※ 要確認(例示値): 効果の定量値は自組織での計測が必要。本スライドの主張は一般論。

まとめ

Claude Code の安全性は『最小権限・隔離・人によるレビュー』の3原則に集約される。

  • 権限: 既定を安全側に。許可制/プランモードを基本にする
  • 隔離: サンドボックスでシークレットと本番から切り離す
  • レビュー: 差分は人が確認し、汚染入力を指示にしない

明日からの一歩

まずは権限とシークレットの初期設定を整え、レビュー手順をチームに明文化する。

  • 1. 権限モードを許可制に設定し、破壊的コマンドの deny ルールを追加
  • 2. .env/secrets を除外設定と .gitignore で読ませない・置かない
  • 3. サンドボックス/コンテナで隔離実行する環境を用意
  • 4. 差分レビュー&コミット前確認の手順をチームで合意・明文化
  • 5. 自動レビュー(security-guidance 等)の導入を評価（要確認: 提供状況）
