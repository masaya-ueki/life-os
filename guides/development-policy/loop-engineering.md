## 1. このガイドの目的

life-os が目指す開発スタイルは **Loop Engineering（loop エンジニア）** である。

Loop エンジニアとは、AI に一手ずつプロンプトを打つ人ではなく、**AI を呼び出す loop（自動化された反復）そのものを設計・所有する人**を指す。コードを書くのも、次に何を作るかを決めるのも loop の内側に委ね、人は loop の外側で「意図」と「歯止め」を設計する側に回る。

このガイドは、Boris Cherny が示した「loop」の三段階定義を要約し、life-os が今どの段階にいて、次に何を積み上げれば Stage 3 に近づけるかを共有することを目的とする。これは ADR（アーキテクチャ決定）ではなく**開発運用の指針**なので `guides/development-policy/` に置く。

---

## 2. 「loop」の三段階定義（Boris Cherny）

Boris Cherny は、自身の働き方の変化を 3 つの段階で説明している。本質は「**ループ（反復）の内側に誰が座っているか**」が段階ごとに変わっていく点にある。

| 段階 | 立ち位置 | 働き方 |
|------|---------|--------|
| **Stage 1（過去）** | モデルは**自分が操作する道具** | 補完を使いながらコードを手で書く。モデルを一行ずつ指示して制御する。 |
| **Stage 2（少し前）** | モデルは**並列に動かす相棒** | 5〜10 個の Claude セッションを並行で走らせる。それでも各セッションに**プロンプトを打つのは常に自分**。 |
| **Stage 3（現在）** | モデルは**自分のプログラムが呼ぶサブルーチン** | 直接プロンプトを打たない。**Claude をプロンプトする loop を書く**。多数のエージェントがリポジトリや各種チャンネルを読み、次に何を作るかを自律的に決める。 |

> Cherny の言葉: 「もう Claude にプロンプトを打っていない。走っている loop がある。Claude にプロンプトを打っているのはその loop だ。」

ポイントは、Stage が進むほど人の関与が**減る**のではなく、関与の**層が上がる**こと。手を動かす対象が「コード」から「セッション群」へ、そして「loop の設計」へと上がっていく。

---

## 3. 核心 — loop の「内側」から「作者」へ

三段階を貫く一つの転換がある。

- **Before**: 自分が「プロンプトを打ち込む、プロセスの内側の存在」
- **After**: 自分が「**loop そのものの作者**」

モデルは「自分が操作する道具」から「**自分のプログラムが呼ぶサブルーチン**」へと役割を変える。だから loop エンジニアの仕事は「良いプロンプトを打つこと」ではなく、

- どんな**意図**を loop に与えるか
- loop が**自分で出力を検証**できるようにするには何が必要か
- loop が暴走しないための**歯止め**をどこに置くか

を設計することになる。プロンプトは loop の内部実装の一部に降りていく。

---

## 4. Loop Engineering の実践原則

記事が挙げる loop engineering の実践は、次の 5 つに整理できる。

1. **意図を一度だけ書く（Write intent once）**
   一手ずつの手順書ではなく、「何を達成したいか」を一度書く。手順への翻訳は loop に任せる。
2. **反復の管理を委譲する（Delegate iteration）**
   cron などのスケジュールで動く自動化に、反復を回す仕事を渡す。人が毎回起動しない。
3. **再利用可能なスキル群を育てる（Reusable skill libraries）**
   loop が繰り返し呼び出せるスキル/手順のライブラリを用意する。loop の語彙が増えるほど任せられる範囲が広がる。
4. **検証ゲートを組み込む（Validation gates）**
   loop が次に進む前に、自分の出力を自分で検証できるようにする。通らなければ進ませない。
5. **ハードストップを置く（Hard stops）**
   反復回数の上限・無進捗の検出・トークン/金額の予算などで、暴走と浪費を止める。

> 5 は特に重要。loop に自律性を与えるほど、止め方を先に設計しておかないとコストが青天井になる。

## 5. life-os は今どこにいるか

life-os は、土台づくりを通じて **Stage 2 を運用しつつ Stage 3 の部品を仕込んでいる**段階にある。実践原則と既存資産は次のように対応する。

| 実践原則 | life-os の現状の資産 |
|---------|----------------------|
| 意図を一度だけ書く | **Issue 駆動**（ProductBacklog / Task Issue）。意図と完了条件を Issue に一度書き、セッションをまたいで引き継ぐ — [Issue 運用ルール](./issue-operation-rules.md) |
| 反復の管理を委譲する | Claude Code ネイティブの複数セッション運用へ移行済み（[ADR-0001](../../docs/adr/0001-claude-code-native-multi-session.md)）。`/loop`・スケジュール実行などのネイティブ機能が委譲の土台 |
| 再利用可能なスキル群 | `.claude/skills/`・`.claude/agents/`（例: スライド生成パイプライン [ADR-0003](../../docs/adr/0003-presentation-system.md)）が loop から呼べる再利用部品 |
| 検証ゲート | `uv run lint-imports`（領域境界）/ `uv run pytest`（スモークテスト）。**Modular Monolith × Bounded Context**（[ADR-0002](../../docs/adr/0002-modular-monolith-bounded-context.md)）が「壊れたら検知できる」境界を機械的に保証する |
| ハードストップ | GitHub Flow + PR レビュー（`main` へのマージ前に人が確認）。当面は PR が最後の歯止め |

つまり life-os は、**Stage 2（並列セッションを人がプロンプトする）を回しながら、Stage 3（loop が自律的に動く）に必要な「意図・スキル・検証・境界」を先に整えている**。

---

## 6. Stage 3 へ近づくための具体ステップ

順に積み上げていく。各ステップは独立しても価値があり、揃うほど loop に任せられる範囲が広がる。

1. **意図を構造化して書く癖をつける**
   Issue に「何を・なぜ・完了条件」を明示する。loop が読んで判断できる粒度で書く。
2. **検証ゲートを増やし、緑を当たり前にする**
   `lint-imports` / `pytest` に加え、領域ごとの受け入れ条件をテスト化していく。loop が「自分で通せる」状態を作る。
3. **スキル/サブエージェントを資産として育てる**
   繰り返す作業を `.claude/skills/`・`.claude/agents/` に切り出す。loop の語彙を増やす。
4. **反復をスケジュールに乗せる**
   定期実行（cron / `/loop` 等）で、人が起動しなくても「読む → 提案する → 検証する」が回る小さな loop から始める。
5. **ハードストップを必ず先に設計する**
   反復上限・無進捗検出・予算を loop の起動条件に組み込む。歯止めのない自律化はしない。

> 重要な順序: **検証ゲートとハードストップ（2・5）が、自律化（4）より先**。止め方と確かめ方が無いまま loop に任せない。

## 7. やらないこと・気をつけること

- **段階の飛ばし読みをしない** — Stage 2 を安定して回せないうちに Stage 3 の自律 loop を作らない。
- **検証なしで loop を走らせない** — 出力を確かめる手段（テスト・lint・レビュー）が無い loop は、間違いを高速に量産する。
- **歯止めを後回しにしない** — トークン/金額の予算・反復上限は「あとで付ける」ではなく起動条件にする。
- **人の判断点を消し切らない** — life-os は個人の暮らしを扱う。重要な意思決定（公開・課金・外部送信・削除など）は loop の外で人が確認する。
