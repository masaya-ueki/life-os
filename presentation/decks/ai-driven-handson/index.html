AI駆動開発のこれから

「AIにコードを書かせる」段階は終わり、「AIに仕事を任せる」段階へ。対象は開発から全業務に広がる。

  • コード生成は当たり前 → AIが「タスクを完遂」する時代へ
  • 適用範囲が開発から、調査・資料作成・運用・定型業務へ拡大
  • 人の役割は「実装者」から「AIに任せる設計者・レビュアー」へシフト
  • 勝ち筋は「仕組み化」: 一度作れば何度でも効く資産を積む

開発スタイルの変遷

手書き → 補完 → エージェント駆動 → loop 駆動。人の関与は「1行ずつ書く」から「流れを設計する」へ移る。

  1. 1

    手書き

    人が全部書く

  2. 2

    補完

    AIが部分提案・人が主役

  3. 3

    エージェント駆動

    AIがタスクを実行・人はレビュー

  4. 4

    loop 駆動

    AIが輪を回す・人は輪を設計

座学3 — 3回やったらAI化

同じ作業を3回やったら、AI化しよう。

手動で繰り返している作業はないか？を常に問う。
「面倒」を感じた瞬間が、AI化のシグナル。

自分は、作業フローの一部

ほぼすべての作業はAIで自動化できる。ただしスキル・ルールの充実があって初めて成立する。

人はフロー設計とレビューに専念
エージェントがタスクを自律実行
土台: ルール / スキル / フックの蓄積

効率化の仕組み = ディレクトリ構成

AIを効率的に使う鍵は、リポジトリ直下に置く「4点セット」。Agent / Skill / Rule / Hook。

.claude/（AIの設定一式）
agents/
役割特化エージェント
skills/
再利用ワークフロー（SKILL.md）
rules/
常時適用ルール
hooks/ + settings
自動ブロック・自動整形
commands/
/learn・/evolve 等

4点セットの役割と実例

それぞれが「いつ効くか」を実物で押さえる。組み合わせでAIの精度と安全性が決まる。

守り（Rule / Hook）

  • Rule: main保護・命名規約・セキュリティ方針
  • Hook: terraform destroy / force-push をブロック
  • Hook: .env 書き込み・秘密情報を検出
×

攻め（Agent / Skill / Command）

  • Agent: 設計 / 実行 / レビューを分業
  • Skill: issue 作成・dbt テスト等の定型手順
  • Command: /learn → /evolve で資産を育てる

何に向くサービスか？

結論、ほとんどの業務はコーディングエージェントで実現できる。4の考え方はツールを問わず共通。

  • 資料作成・調査・データ処理・運用 — 多くは「ファイルとコマンド」に落ちる
  • ファイルとコマンドを扱えるコーディングエージェントは、適用範囲が最も広い
  • Agent / Skill / Rule / Hook の考え方は Claude Code でも Copilot でも共通
  • → まずコーディングエージェントを「最強の基盤」として押さえるべき
エンジニアは、コードを1行ずつ書く個人ではなく
「エージェントのテックリード」として考えるべきだ。

— Sridhar Ramaswamy（Snowflake CEO）

コーディングエージェントを使いこなす人材（uber programmers）は1年前比 50〜100倍の生産性。
鍵はモデルの大きさではなく「システム設計」— まさに4点セットの話。

リポジトリ構成: .claude と .github のミラー

同じ思想を、Claude Code は .claude/、GitHub Copilot は .github/ で表現する。片方を学べば両方に効く。

.claude/（Claude Code）

  • agents/ — 役割特化エージェント
  • skills/ — SKILL.md
  • rules/ — 常時適用ルール
  • hooks/ + settings.json — 自動チェック
  • commands/ — /learn・/evolve
≒

.github/（GitHub Copilot）

  • agents/ — *.agent.md
  • skills/ — 同名スキルをミラー
  • instructions/ — *.instructions.md（≒rules）
  • copilot-instructions.md — 全体方針
  • ISSUE_TEMPLATE/ — issue 雛形

複数回やっている作業・課題を書き出す

loop の燃料は「自分の繰り返し作業」。まず手で書き出して、AI化候補を見つける。

  • 直近で3回以上やった作業をリストアップ
  • 各作業の「手順・入力・出力・つまずき」を一言で添える
  • 「面倒・毎回ググる・コピペ」が多いものほど良い候補
  • 1つ選ぶ — 本日はこれをAI化する

リポジトリをクローン（or fork してpush）

コピー済みハンズオン用リポジトリを手元に。今後を見据え「各自リポジトリを持つ」かはAIと相談。

Claude Code レーン

  • git clone でハンズオン用リポジトリを取得
  • .claude/ の agents・skills・rules・hooks を確認
  • worktree で作業（git.md ルールに従う）
→

GitHub Copilot レーン

  • fork → clone（自分のリポジトリで進める）
  • .github/ の instructions・agents・skills を確認
  • copilot-instructions.md で全体方針を把握

課題整理エージェントで対話 → md 化

（今後作る）課題整理エージェントと対話し、自分の作業を skill / rule / agent / hook のベースとして md に起こす。

  1. 1

    説明

    選んだ作業をエージェントに話す

  2. 2

    深掘り

    手順・前提・禁止事項を質問で引き出す

  3. 3

    分類

    skill / rule / agent / hook のどれかに割り振る

  4. 4

    md 化

    各ベースを Markdown で出力

AIですべて作成

たたき台 md を渡し、AIに skill / rule / agent / hook の本体を一括生成させる。レーンで置き場所が変わるだけ。

Claude Code が生成

  • skills/<name>/SKILL.md
  • rules/<topic>.md
  • agents/<role>.md
  • hooks/*.sh + settings.json
≒

GitHub Copilot が生成

  • skills/<name>/（ミラー）
  • instructions/<topic>.instructions.md
  • agents/<role>.agent.md
  • copilot-instructions.md 追記

デモを実行

作った仕組みを実際に動かす。狙った作業がAIで完遂できるかを、その場で確かめる。

  • 生成したスキル / エージェントを実際に起動して作業させる
  • Hook が意図通りに危険操作をブロックするか確認
  • Rule が守られているか（main で作業していない等）を確認
  • うまくいった点・崩れた点をメモ

改善点を洗い出して issue 起票

デモで見えた粗さを issue にする。これが次の一周の入力になり、精度が上がっていく。

  • デモで崩れた点・あと一歩だった点を具体的に書き出す
  • create-issue スキル / gh issue create でテンプレ起票
  • issue が次の「課題整理（実践3）」の入力になる
  • → 3〜6 が閉じて、loop が回り始める

3→6 の loop が、ゴール

課題整理→生成→デモ→改善 issue を繰り返すほど、精度の高い充実した資産になる。
