# ADR-0012: 資格問題の作成を「スキル＋登録サブエージェント＋整合性テストゲート」で担保する

- **ステータス**: `承認済み`
- **決定日**: 2026-07-02
- **決定者**: masaya_ueki
- **関連タスク**: #96 / #97
- **置き換え**: -

---

## コンテキスト

certification 領域の SnowPro Core 問題は当初 4 ジャンル・12 問しかなく、全ジャンルへ網羅的に拡充する必要が生じた（#96）。問題を都度手書きで足すと、**用途理解型か・4択の誤答に理由があるか・出典が公式一次情報か・スキーマ整合か**といった品質が担保されない。

ここで決めるべき構造的な論点は次の 3 つ。いずれも「一度決めると後から変えにくい」「有力な選択肢が複数あり自明でない」ため ADR の対象（[ADR README の判断基準](./README.md#adr-の必要性判断基準)）に該当する。

1. **作成基盤の構成**: 品質基準と実行をどう分離・再利用可能にするか。
2. **データ整合性の強制場所**: id 重複・正解数・NG理由・出典など、ローダーが見ない不変条件をどこで機械的に守るか。
3. **試験バージョン**: データが載る `COF-C02` は retired（後継 `COF-C03` が 2026-03-30 に live）。どちらを正とするか。

前提として、`adapters/repository.py` のローダーは必須キーの有無と `format` enum しか検証せず、既存テストは件数系スモークのみで、**データ品質を守る仕組みが無い**。また presentation 領域が「索引スキル＋ワーカーサブエージェント」で生成基盤を確立している（[ADR-0003](./0003-presentation-system.md)）。

## 決定事項

**問題作成を、品質基準を索引する `cert-question-authoring` スキルと、生成→検証→登録を担う `snowpro-question-author` サブエージェントに分離する**（ADR-0003 の「方針の索引＝スキル／実行＝サブエージェント」を certification に適用）。**データ整合性は `tests/test_data_integrity.py` の pytest ゲートで強制**し、`docker compose run --rm test` を登録時の検証ゲートにする。**試験バージョンは COF-C03 に完全移行**し、第5ジャンル `data-collaboration` を追加する。

## 検討した選択肢

### データ整合性の強制場所

#### 選択肢A: pytest 整合性ゲート（採用）
- **メリット**: 既存の `docker compose run --rm test` にそのまま乗り、CI/ローカルの検証ゲートと一体化。ドメインのローダーを再利用して「アプリが実際に読む表現」を検査でき、可読な失敗メッセージを出せる。追加依存ゼロ。
- **デメリット**: 実行に pytest/Docker が要る（JSON 単体の即時検査ではない）。サブエージェントはインライン python でも自己検査する運用で補う。

#### 選択肢B: ローダー（repository.py）に検証を追加（不採用）
- **メリット**: 読み込み時に必ず走る。
- **デメリット**: ドメイン/アダプタに検証責務が混入し、単一責務が崩れる。壊れたデータでアプリ全体が起動不能になる（テストの方が影響を局所化できる）。
- **不採用理由**: 整合性は「データ投入時に落とすべき CI 関心事」で、実行時ローダーの責務ではない。

#### 選択肢C: JSON Schema ファイル＋外部バリデータ（不採用）
- **メリット**: 宣言的。
- **デメリット**: jsonschema 等の依存追加と別実行系が要る。id 一意・正解数などの相関制約は結局手続き的コードが要り、二重管理になる。
- **不採用理由**: 依存と実行系を増やす割に、pytest で書ける以上の利点が無い。

### 作成基盤の構成

#### 選択肢A: 索引スキル＋ワーカーサブエージェント（採用）
- **メリット**: ADR-0003 で確立済みの分離を再利用でき、方針（品質基準・採番・被覆表）と実行（生成・登録）を独立に改善できる。他資格へ横展開しやすい。
- **デメリット**: ファイルが分散する（スキル本体＋references＋エージェント）。

#### 選択肢B: 単一のプロンプト/スクリプトで一括生成（不採用）
- **デメリット**: 品質基準が実行に埋もれ再利用・改善がしづらい。
- **不採用理由**: 「全ジャンルを回して基盤を改善し続ける」ループ（#98）に対し、方針と実行が癒着すると改善の当て先が曖昧になる。

### 試験バージョン

#### 選択肢A: COF-C03 へ完全移行（採用）
- **メリット**: live 試験（最新の 5 ドメイン構成）に整合。学習価値が高い。
- **デメリット**: 既存4ジャンルの `name` 更新と新ジャンル追加が必要。

#### 選択肢B: COF-C02 を維持（不採用）
- **不採用理由**: C02 は retired。第5ドメイン Data Collaboration を欠いたままとなり「全ジャンル網羅」を満たせない。

## 結果・トレードオフ

- 問題は品質基準（用途理解型・NG理由・公式出典）を満たす形でのみ増える。破れば `test` が赤になる。
- ジャンルの `id` は据え置き（`name` のみ更新）としたため、既存問題の `genre_id`・id 接頭辞は無傷。既存12問は保持。
- 網羅性は「件数の下限」ではなく **genre-doc-map の被覆表**で測る（#98）。
- 将来、他資格を足す場合は `genres`・接頭辞・被覆表を拡張し、同じスキル／エージェント／整合性ゲートを再利用する。
