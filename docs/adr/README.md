# Architecture Decision Records (ADR)

このディレクトリは、プロジェクトの重要な設計決定とその理由を記録します。

---

## ADR とは

ADR（Architecture Decision Record）は「なぜその設計にしたのか」を記録するドキュメントです。

設計の **What（何を）** はコードや設計ドキュメントに書かれていますが、**Why（なぜ）** は時間が経つと失われがちです。ADR はその「なぜ」を残すことで、将来の変更判断や新規参加者のオンボーディングを助けます。

## ADR の必要性判断基準

**すべての設計に ADR は不要です。** 以下のいずれかに該当する場合のみ ADR を作成してください。

### ADR が必要な設計

| 基準 | 例 |
|------|----|
| **一般的な慣習や標準から外れた選択をした** | 標準的な構成をあえて採用しない |
| **複数の有力な選択肢が存在し、採用理由が自明でない** | データ保存形式の選定、ライブラリ選定 |
| **将来の変更・拡張に影響する構造的な判断** | ディレクトリ設計、モジュール分割方針 |
| **「なぜこうなっているのか？」と疑問を持たれる可能性が高い** | 独自ルール・しきい値の設定 |
| **一度決めると変更コストが高い判断** | データ構造、認証方式 |

### ADR が不要な設計

| 基準 | 例 |
|------|----|
| **業界標準・公式ベストプラクティスに従っているだけ** | 変数名は snake_case |
| **唯一の合理的な選択肢しかない** | `.gitignore` に一時ファイルを含める |
| **一時的・実験的な判断（短期間で見直し予定）** | 暫定的な命名、試験的な機能フラグ |
| **コードを読めば意図が明確なもの** | シンプルな変数定義 |

## ADR の作成手順

1. **[template.md](./template.md) をコピー**して、連番のファイル名で保存
   ```
   docs/adr/XXXX-{kebab-case-title}.md
   ```

2. **GitHub Issues でチームに議論を投げる**
   - 複数の選択肢を提示し、意見を収集する

3. **議論の結果を ADR に記録し、ステータスを `提案中` → `承認済み` に変更**

4. **対応する設計ドキュメント・ルールファイルに ADR へのリンクを追記**（必須）

5. **コード変更 PR に ADR ファイルを含める**（コードと同時レビュー）

6. **本 README の「ADR 一覧」表に追記する**

## ADR と設計ドキュメントのリンクルール

**ADR に対応する設計が存在する場合、設計側に必ず ADR へのリンクを張ること。**

### 設計ドキュメント（Markdown）へのリンク例

```markdown
## データ保存形式

> **設計根拠**: [ADR-0001 データ保存形式の選定](../../docs/adr/0001-data-storage-format.md)

タスクデータは JSON 形式で保存する。
```

### コードへのリンク例

```python
# データ保存形式: ADR-0001 参照
# docs/adr/0001-data-storage-format.md
```

## ADR のステータス管理

| ステータス | 意味 |
|-----------|------|
| `提案中` | 議論中・まだ確定していない |
| `承認済み` | チームで合意し、現在適用中 |
| `非推奨` | まだ使われているが、新規採用は推奨しない |
| `置き換え済み` | 別の ADR に置き換えられた（リンク先を記載） |

### 決定が覆った場合の対応

- 古い ADR のステータスを `置き換え済み` に変更し、新しい ADR へのリンクを追記する
- **古い ADR は削除しない**（過去の意思決定の履歴として残す）
