# ADR-0008: PR の自動マージ/人間レビューをパスベースのスコープゲートで判定する

- **ステータス**: `承認済み`
- **決定日**: 2026-06-21
- **決定者**: masaya_ueki
- **関連タスク**: （任意）

---

## コンテキスト

life-os には既に PR レビュー基盤（[ADR-0004](./0004-pr-review-agent.md)）があり、`pr-reviewer` エージェントは
`[must]`=0 ∧ pytest ∧ lint-imports pass を満たせば無人でマージする。しかし「**どの PR を無人マージしてよいか**」
の境界は明文化されておらず、検証ゲートを通れば領域横断・基盤・方針変更でも自動マージされうる状態だった。

運用上「自動マージしてよい改修」と「人間レビューを挟むべき改修」を分けたい。望ましい方針は次の4つ：

1. 各領域に閉じた改修 → 自動
2. 領域をまたがる改修 → 人間
3. 方針にかかわる改修 → 人間
4. バグ修正・ディレクトリ品質改善など簡易なタスク → 自動

ただし**基準が不明瞭になることを避けたい**。基準1〜3は影響範囲（ブラスト半径）で決まり変更パスから機械的に
判定できるが、基準4は「簡易かどうか」という意図ベースで、基準1〜2と衝突する（例: 領域をまたぐバグ修正、
方針ファイルの誤字修正）。この衝突をどう解消し、判定をどこまで決定的にできるかが論点。

## 決定事項

PR の auto / human を、**変更ファイルパスだけで決定的に判定するスコープゲート**で分類する。
判定は「① 方針・境界・契約パスを含む → human」「② shared を含む → human」「③ Bounded Context を2つ以上含む
→ human」「④ いずれにも非該当（単一領域 or content のみ）→ auto」の順で評価する。
基準4（簡易タスク）は独立トリガーにせず**スコープ判定に従属させる**（単一領域内なら auto、横断なら human）。
判定ロジックは [`review-and-merge-pr`](../../.claude/skills/review-and-merge-pr/SKILL.md) スキルを単一の真実とし、
レビュー観点・検証ゲート・マージ手順は既存資産（`code-review-*`・`pr-reviewer`）を再利用する。
shared は単一ディレクトリだが全領域に波及するため human、auto の最終マージは完全無人とする。

## 検討した選択肢

### 選択肢A: パスベースのスコープゲート＋基準4をスコープに従属（採用）

- **メリット**: 判定が変更パスのみで決定的（曖昧さゼロ）。基準1〜3を素直に表現でき、基準4の衝突も
  「スコープが安全だから auto」に一元化することで解消できる。既存レビュー基盤を再利用でき追加実装が薄い。
- **デメリット**: directory-keeper の横断的な品質改善（README 雛形・リネーム・リンク修正）は③/①で human になり、
  「簡易なのに自動にならない」ケースが残る。

### 選択肢B: 基準4を意図ベースの独立トリガーにする（不採用）

- **メリット**: 「バグ修正・品質改善は領域横断でも自動」という基準4を文字どおり満たせる。
- **デメリット**: 「簡易/バグ修正かどうか」を機械判定できず、判定が主観的・非決定的になる。
- **不採用理由**: ユーザーの最重要要件「基準を不明瞭にしない」に反する。基準1〜2とも直接衝突する。

### 選択肢C: スコープゲートを設けず検証ゲートのみで自動マージ（現状維持）（不採用）

- **メリット**: 仕組みを追加せずに済む。
- **デメリット**: 領域横断・基盤・方針変更でも pytest/lint が通れば無人マージされ、影響範囲の大きい変更が
  人手を経ずに main へ入る。
- **不採用理由**: 「方針・横断は人間」という要件を満たせない。

## 結果・トレードオフ

- auto / human が変更パスから一意に決まり、レビュー運用の予測可能性が上がる。判定理由（①〜④）を提示できる。
- shared と方針/境界/契約ファイルは常に human に倒れ、ブラスト半径の大きい変更が無人マージされない。
- トレードオフ: 横断的な品質改善（directory-keeper 由来など）は human レビューを要する。これは「スコープ厳格」を
  優先した結果であり、将来 directory-keeper 由来の機械的修正を安全に識別できるシグナル（専用ラベル等）が
  整えば、④の例外として auto を許可する再検討の余地がある（その際は本 ADR を更新する）。
- 検証はローカル/Docker 実行に依存する（CI 未設置）。CI 導入時はゲートを CI 結果に置き換える（ADR-0004 と同様）。
